# PostMaker ‚Äî API Contract (source of truth for frontend2)

Generated by reading `frontend/src/lib/api.ts` + every actual call site in
`pages/` and `components/`. This is what frontend2 must be built against ‚Äî
no endpoint here should be reinvented, and no new endpoint should be
assumed unless it's listed in "NOT YET BUILT" at the bottom.

## Global rules (apply to every call)
- Base path: `/api`
- **Auth is cookie-based.** Every request must send `credentials: 'include'`.
  There is no token to attach manually ‚Äî forgetting this option makes every
  request look logged-out.
- Standard JSON requests use `Content-Type: application/json`.
- Errors: non-2xx responses return `{ error: string }`. Frontend must throw/
  catch a typed error that preserves the HTTP status code (e.g. 409 =
  already subscribed, 401 = not authenticated) ‚Äî don't collapse all errors
  into one generic message.

## Generate ‚Äî ‚öÝÔ∏è SPECIAL: this is a streaming SSE connection, not a normal fetch
`POST /generate` ‚Äî body `{prompt, platforms: string[], imageKey, hasVideo, videoName}`

Do **not** call this with a plain `fetch().then(res => res.json())`. The
response body is a stream of `event: ... \n data: ...\n\n` frames that must
be read incrementally via `res.body.getReader()` and decoded as they
arrive, firing an `onEvent(event, data)` callback per frame. This is what
powers the "live generation" UX ‚Äî implementing it as a normal request will
make the UI hang until the entire generation finishes instead of showing
progress.

| Action | Call | Returns |
|---|---|---|
| Retry one platform | `POST /generate/retry` | body `{campaignId, platformId}` ‚Üí `{content, platformId}` |

## History
| Action | Call | Returns |
|---|---|---|
| List campaigns | `GET /history?page=N` | `{campaigns: [], pagination: {}}` |

‚öÝÔ∏è **Known gap:** campaigns in the DB do not store an image/video URL ‚Äî
only `has_image`/`has_video` boolean flags. Thumbnails shown in "My
Generations" / "Recent Generations" designs **cannot** be populated from
this endpoint until the backend is extended (see gaps section below).

## Payments / Billing
| Action | Call | Returns |
|---|---|---|
| Get currency by geo | `GET /payments/currency` | `{currency, country}` |
| Get subscription status | `GET /payments/status` | `{user, subscription}` |
| Get status by sub id | `GET /payments/status/:id` | `{subscription: {plan, status, current_period_end, currency} \| null}` |
| Subscribe | `POST /payments/subscribe` | body `{plan, currency, promoCode?}` ‚Üí `{subscriptionId, keyId, plan, currency}` |
| Cancel | `POST /payments/cancel` | ‚Äî |

‚öÝÔ∏è **Sequencing dependency:** `subscribe` requires `currency` to already be
resolved via `payments.currency()` first. Don't hardcode USD.

## Upload ‚Äî ‚öÝÔ∏è SPECIAL: two-step, not one call
1. `POST /upload/presign` ‚Äî body `{contentType, contentLength}` ‚Üí `{uploadUrl, objectKey}`
2. Frontend must then `PUT` the actual file **directly to `uploadUrl`**
   (not to the API). Skipping step 2 means "upload" silently does nothing.
3. `objectKey` from step 1 is what gets passed as `imageKey` into `/generate`.

## Brand Kit
| Action | Call | Method | Body | Returns |
|---|---|---|---|---|
| Get Brand Kit | `GET /api/brand-kit` | GET | ‚Äî | `{ brandKit: { name, logo_object_key, logo_dark_key, logo_icon_key, colors, fonts, voice, social_links, platform_links, products_services, target_audience, competitors, brand_guidelines, updated_at } }` |
| Save Brand Kit | `PUT /api/brand-kit` | PUT | `{ name, logo_object_key, logo_dark_key, logo_icon_key, colors, fonts, voice, platform_links, products_services, target_audience, competitors, brand_guidelines }` | `{ ok: true, message: 'Brand kit saved successfully' }` |
| Delete Brand Kit | `DELETE /api/brand-kit` | DELETE | ‚Äî | `{ ok: true, message: 'Brand kit deleted successfully' }` |
| Stream Brand Asset | `GET /api/brand-kit?assetKey=uploads/...` | GET | ‚Äî | Streams R2 object binary body (200 OK) if `assetKey` matches user's own stored logo key (`logo_object_key`/`logo_dark_key`/`logo_icon_key`); returns 404 error JSON otherwise. |

## NOT YET BUILT ‚Äî endpoints referenced by the new designs but absent from both API and DB schema

These pages/features cannot be wired to real data yet. Building their UI
now means either (a) shipping a static/mocked screen, or (b) building the
backend endpoint + DB table first. Do not silently invent a fake call that
looks real ‚Äî mark these components clearly as mocked until backend exists.

| Feature | Missing piece |
|---|---|
| Dashboard "Recent Generations" thumbnails | `campaigns` table has no `image_url`/`image_key` persisted after generation ‚Äî image is currently discarded |
| Generation Details page (view original image) | same as above |
| "Create Similar" (reuse original image) | same as above |
| My Generations status/thumbnails | same as above |
| Templates page | no `templates` table, no CRUD endpoints |
| Notifications bell | no `notifications` table, no list/mark-read endpoints |
| Analytics ‚Äî Total Downloads, Top Platforms breakdown | `usage` table only tracks generation counts per period; no per-platform or download tracking |
| Global search ("Search anything...") | no search endpoint |
| Documentation page | no content source ‚Äî needs to be authored, not an API gap |

`api_keys` table already exists in the schema (marked "v2 foundation") but
has no endpoints yet ‚Äî relevant if/when a public API product is built.
