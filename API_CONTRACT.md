# PostMaker — API Contract (source of truth for frontend2)

Generated by reading `frontend/src/lib/api.ts` + every actual call site in
`pages/` and `components/`. This is what frontend2 must be built against —
no endpoint here should be reinvented, and no new endpoint should be
assumed unless it's listed in "NOT YET BUILT" at the bottom.

## Global rules (apply to every call)
- Base path: `/api`
- **Auth is cookie-based.** Every request must send `credentials: 'include'`.
  There is no token to attach manually — forgetting this option makes every
  request look logged-out.
- Standard JSON requests use `Content-Type: application/json`.
- Errors: non-2xx responses return `{ error: string }`. Frontend must throw/
  catch a typed error that preserves the HTTP status code (e.g. 409 =
  already subscribed, 401 = not authenticated) — don't collapse all errors
  into one generic message.

## Auth
| Action | Call | Method | Body | Returns |
|---|---|---|---|---|
| Google OAuth | `GET /auth/google` | GET | — | redirects to Google |
| Logout | `POST /auth/logout` | POST | — | — |
| Email signup | `POST /auth/email/signup` | POST | `{name, email, password}` | `{ok}` |
| Email login | `POST /auth/email/login` | POST | `{email, password}` | `{ok}` |
| Resend verification email | `POST /user/resend-verification` | POST | — | `{ok}` |
| Forgot password | `POST /auth/email/forgot-password` | POST | `{email}` | `{ok, message}` |
| Reset password | `POST /auth/email/reset-password` | POST | `{email, token, password}` | `{ok}` |

## User
| Action | Call | Returns |
|---|---|---|
| Get current user + usage | `GET /user/me` | `{user, usage}` |
| Set currency | `PUT /user/currency` | body `{currency: 'usd'|'inr'}` |
| Delete account | `DELETE /user/account` | body `{confirmation}` |

## Generate — ⚠️ SPECIAL: this is a streaming SSE connection, not a normal fetch
`POST /generate` — body `{prompt, platforms: string[], imageKey, hasVideo, videoName}`

Do **not** call this with a plain `fetch().then(res => res.json())`. The
response body is a stream of `event: ... \n data: ...\n\n` frames that must
be read incrementally via `res.body.getReader()` and decoded as they
arrive, firing an `onEvent(event, data)` callback per frame. This is what
powers the "live generation" UX — implementing it as a normal request will
make the UI hang until the entire generation finishes instead of showing
progress.

| Action | Call | Returns |
|---|---|---|
| Retry one platform | `POST /generate/retry` | body `{campaignId, platformId}` → `{content, platformId}` |

## Refine
| Action | Call | Returns |
|---|---|---|
| Refine one post | `POST /refine` | body `{campaignId, platformId, message, currentContent}` → `{content, platformId}` |

## History
| Action | Call | Returns |
|---|---|---|
| List campaigns | `GET /history?page=N` | `{campaigns: [], pagination: {}}` |

⚠️ **Known gap:** campaigns in the DB do not store an image/video URL —
only `has_image`/`has_video` boolean flags. Thumbnails shown in "My
Generations" / "Recent Generations" designs **cannot** be populated from
this endpoint until the backend is extended (see gaps section below).

## Payments / Billing
| Action | Call | Returns |
|---|---|---|
| Get currency by geo | `GET /payments/currency` | `{currency, country}` |
| Get subscription status | `GET /payments/status` | `{user, subscription}` |
| Get status by sub id | `GET /payments/status/:id` | `{subscription: {plan, status, current_period_end, currency} \| null}` |
| Subscribe | `POST /payments/subscribe` | body `{plan, currency, promoCode?}` → `{subscriptionId, keyId, plan, currency}` |
| Cancel | `POST /payments/cancel` | — |

⚠️ **Sequencing dependency:** `subscribe` requires `currency` to already be
resolved via `payments.currency()` first. Don't hardcode USD.

## Upload — ⚠️ SPECIAL: two-step, not one call
1. `POST /upload/presign` — body `{contentType, contentLength}` → `{uploadUrl, objectKey}`
2. Frontend must then `PUT` the actual file **directly to `uploadUrl`**
   (not to the API). Skipping step 2 means "upload" silently does nothing.
3. `objectKey` from step 1 is what gets passed as `imageKey` into `/generate`.

## NOT YET BUILT — endpoints referenced by the new designs but absent from both API and DB schema

These pages/features cannot be wired to real data yet. Building their UI
now means either (a) shipping a static/mocked screen, or (b) building the
backend endpoint + DB table first. Do not silently invent a fake call that
looks real — mark these components clearly as mocked until backend exists.

| Feature | Missing piece |
|---|---|
| Dashboard "Recent Generations" thumbnails | `campaigns` table has no `image_url`/`image_key` persisted after generation — image is currently discarded |
| Generation Details page (view original image) | same as above |
| "Create Similar" (reuse original image) | same as above |
| My Generations status/thumbnails | same as above |
| Templates page | no `templates` table, no CRUD endpoints |
| Brand Kit page | no `brand_kit_assets` table, no CRUD endpoints |
| Notifications bell | no `notifications` table, no list/mark-read endpoints |
| Analytics — Total Downloads, Top Platforms breakdown | `usage` table only tracks generation counts per period; no per-platform or download tracking |
| Global search ("Search anything...") | no search endpoint |
| Documentation page | no content source — needs to be authored, not an API gap |

`api_keys` table already exists in the schema (marked "v2 foundation") but
has no endpoints yet — relevant if/when a public API product is built.
